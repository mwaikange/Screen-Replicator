## PART A — API DOCUMENTATION CORRECTIONS

The official API documentation (v2.0.0) contains the following errors that will cause silent failures or database constraint violations if followed as written. Fix your code against these corrections before touching anything else.

### A1. `incident_types` — columns are wrong in API doc

| What API doc says | Actual column name | Impact |
|---|---|---|
| `incident_types.name` | `incident_types.label` | Queries return null for type display |
| `incident_types.color` | **Does not exist** | This column is not in the DB |
| `incident_types.icon` | **Does not exist** | This column is not in the DB |
| `incident_types.id` | ✅ Correct | — |

**Actual `incident_types` columns:** `id` (serial), `code` (text), `label` (text), `severity` (int)

**Actual seed data (8 types):**
```
code: 'ALERT'      label: 'Emergency Alert'       severity: 5
code: 'CRIME'      label: 'Crime Report'           severity: 4
code: 'GBV'        label: 'Gender-Based Violence'  severity: 5
code: 'FIRE'       label: 'Fire Emergency'         severity: 5
code: 'MEDICAL'    label: 'Medical Emergency'      severity: 5
code: 'MISSING'    label: 'Missing Person'         severity: 4
code: 'SUSPICIOUS' label: 'Suspicious Activity'    severity: 2
code: 'LOST'       label: 'Lost & Found'           severity: 1
```

**Severity-based pin colour** (since there is no `color` column, derive from severity):
```typescript
function getSeverityColor(severity: number): string {
  if (severity >= 5) return '#EF4444';  // Critical — red
  if (severity >= 4) return '#F97316';  // High — orange
  if (severity >= 3) return '#EAB308';  // Medium — yellow
  return '#BEF264';                     // Low — light green
}
```

**Every query that selects `incident_types(id, name, color, icon)` must become:**
```javascript
incident_types(id, code, label, severity)
```

### A2. `incidents` — reaction count columns are wrong in API doc

API doc section 4.1 lists: `upvotes, downvotes, love_count, confirm_count`  
**Actual columns added by migrations:** `upvotes` (int), `downvotes` (int), `loves` (int)  
**`love_count` and `confirm_count` do not exist.**

However, **the actual Vercel feed does not select these columns at all.** It fetches reactions from the `incident_reactions` table separately. The mobile app should do the same.

**Reaction counts — correct approach:**
```javascript
// Batch-load reactions for all posts in feed:
const { data: reactions } = await supabase
  .from('incident_reactions')
  .select('incident_id, reaction_type, user_id')
  .in('incident_id', postIds);

// Count them client-side:
reactions.forEach(r => {
  if (r.reaction_type === 'upvote')   counts[r.incident_id].upvotes++;
  if (r.reaction_type === 'downvote') counts[r.incident_id].downvotes++;
  if (r.reaction_type === 'love')     counts[r.incident_id].loves++;
  if (r.reaction_type === 'confirm')  counts[r.incident_id].confirms++;
});
```

**Valid reaction_type values (DB CHECK constraint):** `'upvote'`, `'downvote'`, `'love'`, `'confirm'`  
**Invalid (will fail with constraint error):** `'helpful'`, `'not_helpful'`, `'verified'`

### A6. `notifications` — two tables exist, each with a different purpose

This is the most confusing part of the codebase. Both tables exist and are used:

| Table | Written by | Read by | Schema |
|---|---|---|---|
| `notifications` | `followUser()` action | Nothing in current Vercel UI | `id, user_id, type, metadata (jsonb), is_read (bool), created_at` |
| `user_notifications` | DB triggers (follow, group events) | Notifications page | `id, user_id, type, title, message, entity_id (uuid), read_at (timestamp), created_at` |

**The mobile app Notifications screen must use `user_notifications`.**  
The `notifications` table (old one) is a legacy write-only table — do not read from it.

**`user_notifications` unread check uses `read_at IS NULL`** — there is no `is_read` column.

**Valid `user_notifications.type` values (DB CHECK constraint):**  
`'follow'`, `'group_request'`, `'subscription'`, `'system'`

API doc section 11 incorrectly says `type: 'new_follower'`, `'incident_comment'`, `'incident_reaction'`, `'group_request_approved'`, `'group_message'`, `'subscription_expiring'` — **none of these exist in the `user_notifications` table**.

**Correct type → navigation mapping:**
```typescript
switch (notification.type) {

  case 'group_request':
    navigation.navigate('Groups'); break;   // entity_id = group_id
  case 'subscription':
    navigation.navigate('Subscribe'); break;
  case 'system':
    break;  // informational only
}
```

**Real-time subscription for notifications must also use `user_notifications`:**

## PART B — STORAGE INTEGRATION

All file uploads go through the Vercel proxy endpoint. Never call Vercel Blob directly from mobile.

### Upload endpoint:
```
POST https://app.ngumus-eye.site/api/upload
Content-Type: multipart/form-data
Field: "file"  (singular)
```
Returns: `{ "url": "https://public.blob.vercel-storage.com/..." }`

### URL storage by upload type:
| Upload | Table | Column |
|---|---|---|
| Incident photo/video | `incident_media` | `path` (stores full URL) |
| Comment photo | `comments` | `image_url` |
| Case evidence | `cases` | `evidence` (JSONB array, each item has `file_url`) |
| Case documents | `cases` | `documents` (JSONB array, each item has `file_url`) |
| Group chat image | `group_messages` | `image_url` |
| Group chat video | `group_messages` | `video_url` |
| Profile avatar | `profiles` | `avatar_url` |

### Replit server fix (replace `/api/upload` with Vercel Blob):
```typescript
// server/routes.ts — replace diskStorage with memoryStorage + @vercel/blob
import { put } from '@vercel/blob';

const upload = multer({
  storage: multer.memoryStorage(),
  limits: { fileSize: 50 * 1024 * 1024 },
  fileFilter: (_req, file, cb) => {
    const allowed = /\.(jpg|jpeg|png|gif|mp4|mov|webp)$/i;
    allowed.test(path.extname(file.originalname)) ? cb(null, true) : cb(new Error('Invalid file type'));
  },
});

app.post('/api/upload', upload.single('file'), async (req, res) => {
  try {
    const file = req.file;
    if (!file) return res.status(400).json({ error: 'No file provided' });
    const maxSize = file.mimetype.startsWith('video/') ? 50 * 1024 * 1024 : 10 * 1024 * 1024;
    if (file.size > maxSize) return res.status(400).json({ error: `File too large` });
    const ext = path.extname(file.originalname);
    const blob = await put(`uploads/${randomUUID()}${ext}`, file.buffer, {
      access: 'public',
      contentType: file.mimetype,
      token: process.env.BLOB_READ_WRITE_TOKEN,
    });
    res.json({ url: blob.url });  // singular 'url', not 'urls' array
  } catch (e: any) {
    res.status(500).json({ error: e.message || 'Upload failed' });
  }
});
```

Add `BLOB_READ_WRITE_TOKEN` to Replit Secrets (get value from Vercel project dashboard → Settings → Environment Variables).

function getImageUrl(pathOrUrl: string | null): string | null {
